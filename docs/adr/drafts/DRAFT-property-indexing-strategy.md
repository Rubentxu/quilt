# DRAFT ADR: Property Indexing Strategy — VIRTUAL Generated Columns

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-03
- **Source:** Q001-P1 (Pass 1)

## Context

Quilt stores block properties as a JSON blob in the `blocks.properties` column. As the property query UX is built, filtering and sorting blocks by property values (e.g., "find all blocks where priority = A AND status = draft") requires scanning and parsing JSON for every row — an O(n) operation that degrades with graph size.

ADR-0007 established that properties are block-attached, not independent entities. A dedicated `block_properties` table (EAV pattern) was considered and rejected because:
- It contradicts the property-based model settled by ADR-0007
- EAV is a well-known relational anti-pattern (JOIN complexity, type-safety loss, array-value problems)
- It requires data migration from the existing JSON blob

## Decision

**Keep the `properties` JSON blob as the canonical source of truth. Add SQLite `GENERATED ALWAYS AS ... VIRTUAL` columns on the `blocks` table for performance-critical properties, with composite indexes.**

```sql
ALTER TABLE blocks ADD COLUMN priority_virtual TEXT 
    GENERATED ALWAYS AS (json_extract(properties, '$.priority')) VIRTUAL;
ALTER TABLE blocks ADD COLUMN due_date_virtual TEXT 
    GENERATED ALWAYS AS (json_extract(properties, '$.due_date')) VIRTUAL;
ALTER TABLE blocks ADD COLUMN marker_virtual TEXT 
    GENERATED ALWAYS AS (json_extract(properties, '$.marker')) VIRTUAL;
ALTER TABLE blocks ADD COLUMN template_virtual TEXT 
    GENERATED ALWAYS AS (json_extract(properties, '$.template')) VIRTUAL;
ALTER TABLE blocks ADD COLUMN status_virtual TEXT 
    GENERATED ALWAYS AS (json_extract(properties, '$.status')) VIRTUAL;

CREATE INDEX idx_blocks_priority ON blocks(priority_virtual);
CREATE INDEX idx_blocks_due_date ON blocks(due_date_virtual);
CREATE INDEX idx_blocks_marker ON blocks(marker_virtual);
CREATE INDEX idx_blocks_template ON blocks(template_virtual);
CREATE INDEX idx_blocks_status ON blocks(status_virtual);
```

**VIRTUAL (not STORED):** VIRTUAL columns are computed on read with zero storage overhead. SQLite materializes the computed value into index B-trees, so indexed lookups are fast. STORED columns duplicate data and are only warranted when the computed value is read frequently without an index.

**Incremental rollout:** Only add VIRTUAL columns for properties that actually need indexed comparison queries. Free-form custom properties stay in JSON only — no schema bloat.

## Why

1. **ADR-0007 alignment**: Properties are block-attached. No entity tables, no EAV, no fragmentation.
2. **No migration**: Additive change. Existing data and code paths unaffected. JSON blob remains the single source of truth.
3. **Indexable with zero storage**: VIRTUAL columns incur no storage cost but produce indexable values. SQLite's query planner uses these indexes for `WHERE`, `ORDER BY`, and `GROUP BY` on the extracted property.
4. **EAV anti-pattern avoided**: No JOIN complexity, no type-safety loss, arrays handled naturally in JSON.
5. **Proven pattern**: SQLite's JSON functions + generated columns is a well-established pattern for indexing JSON documents in SQLite.

## Consequences

### Positive
- Property queries use indexes instead of full-table JSON scans
- No schema migration — additive and backward-compatible
- JSON remains flexible for custom/free-form properties
- Aligned with existing ADRs and domain model

### Negative
- VIRTUAL column definitions must be manually maintained per property
- Property renames require `ALTER TABLE` to update column expressions
- Multi-value properties (arrays) still require `json_extract()` in WHERE clauses (may need expression indexes as follow-up)
- Adding a new indexed property requires a schema change (though additive)

### Trade-offs accepted
- **VIRTUAL over STORED**: Read-time computation traded for zero storage. Acceptable because indexes handle the hot path.
- **Manual column management**: Accepted until V2 when auto-generation from `BUILTIN_PROPERTIES` can be explored.
- **Array properties deferred**: Multi-value property indexing (e.g., `tags:: [rust, wasm]`) remains on JSON path. Follow-up question Q001-F5.
