# DRAFT ADR: React Framework Selection — Vite + TanStack Router

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-05-30
- **Source:** Q001-P1 (Pass 1)

## Context

Quilt is migrating its frontend from Leptos 0.8 to React (see DRAFT-hybrid-wasm-boundary). The application is CSR-only — ADR-0005 removed Tauri and established client-side rendering. The new frontend must:

- Load and execute Rust/WASM outliner core modules
- Provide type-safe routing with URL-serializable block navigation state
- Support code-splitting at feature boundaries
- Run in Vite dev mode with hot module replacement and WASM module loading

Two meta-frameworks were evaluated: **Vite** (industry-standard React bundler) and **Next.js** (full-stack React framework with SSR/SSG/ISR).

## Decision

**Vite + React with TanStack Router. CSR-only.**

| Component | Selection | Justification |
|-----------|-----------|---------------|
| Bundler | Vite | Fast HMR, WASM via vite-plugin-wasm, standard for CSR React |
| Router | TanStack Router | Type-safe, nested layouts, search param serialization, first-class TypeScript |
| WASM loading | vite-plugin-wasm | Native WASM module loading in Vite dev and build |
| React plugin | @vitejs/plugin-react | Fast Refresh, JSX transform |

**Code-split boundaries (3 points):**
1. Outliner (~150KB WASM) — lazy loaded on document open
2. Editor (~120KB — Tiptap + ProseMirror) — lazy loaded on block focus
3. Journals + Sidebar (~80KB) — lazy loaded on navigation

**Dev proxy**: Vite dev server proxies `/api/*` to Rust backend (quilt-server on port 3001).

**Next.js rejected**: SSR/SSG/ISR capabilities are unnecessary for a CSR-only desktop app with no SEO requirements. Next.js adds complexity (server components, routing conventions, build output) without compensating benefit.

## Why

1. **CSR-only eliminates Next.js advantage**: The app runs in a browser/webview with no server-side rendering needs. Next.js's primary value (SSR/SSG/ISR) is irrelevant. Vite provides the same React development experience with less framework overhead.

2. **TanStack Router > React Router for this use case**: Quilt's block-based navigation requires URL-serializable state (block ID, page, journal date, search filters). TanStack Router provides type-safe search params, nested layouts, and file-based routing without Next.js. React Router's type safety is weaker for search params.

3. **vite-plugin-wasm is proven**: Multiple production Rust/WASM + React apps use vite-plugin-wasm for module loading. Vite's plugin ecosystem provides first-class WASM support without Trunk.

4. **Code-splitting reduces initial load**: The outliner WASM module (~150KB) is only needed when a document is opened. Tiptap editor (~120KB) is only needed when a block is focused. Lazy loading at these boundaries keeps the initial shell under 100KB.

## Consequences

- **Build tooling**: Trunk → Vite. Cargo still builds WASM; Vite bundles TypeScript/React.
- **New dependencies**: vite, @vitejs/plugin-react, vite-plugin-wasm, @tanstack/router-vite, @tanstack/react-router
- **ADR-0005 (Leptos CSR)**: Superseded. Vite + React replaces Leptos + Trunk.
- **URL conventions**: Must be defined — block-level navigation patterns (`/page/:id/block/:blockId`) with search params for filters/state.
- **WASM loading**: Shared memory considerations for code-split WASM modules (instantiation, memory sharing between chunks).
