# DRAFT ADR: State Management — WASM Canonical State

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-05-30
- **Source:** Q006-P1 (Pass 1)

## Context

Quilt's outliner core (~3,692 lines of Rust/WASM) manages the canonical domain state: block CRUD, tree traversal, properties, journal navigation, and reference resolution. This code is tested, framework-independent, and will remain in Rust/WASM after the React migration (see DRAFT-hybrid-wasm-boundary).

In a React application, state management is typically delegated to a JavaScript library (Redux, Zustand, Jotai, MobX). However, Quilt's architecture has the canonical state already in WASM — adding a JS state library creates two competing authorities over the same domain data.

The patterns are:
- **Domain state**: blocks, pages, journals, properties, references — canonical in WASM
- **UI state**: panel visibility, scroll position, active tab, selection — ephemeral, React-managed

## Decision

**NO JavaScript state management library for domain state. Rust/WASM outliner core IS the canonical state. React is a thin rendering layer.**

| Pattern | Implementation | Rationale |
|---------|---------------|-----------|
| Read state | `wasm.get_state()` → returns snapshot | Direct binding, no intermediate JS state layer |
| Mutate state | `wasm.dispatch(command)` → applies mutation | Command pattern (CQRS-lite), WASM is the authority |
| UI-local state | React context / local state | Panels, scroll, tabs — ephemeral, no JS library needed |
| Editor state | Tiptap (ProseMirror) | Tiptap is a VIEW — its internal state is subservient to WASM commands |

**Deleted**: All JS state library candidates (Zustand, Redux, Jotai, MobX). None are used for domain state.

## Why

1. **Inner platform anti-pattern**: When canonical state exists in WASM, adding a JS state library creates two sources of truth for the same data. Synchronizing them introduces a distributed consistency problem (latency, stale data, conflict resolution) where none should exist. The outliner core already has `get_state()` and `dispatch(command)` — these ARE the state management API.

2. **The outliner core IS the state machine**: It has getters, setters, commands, and validation logic. Wrapping it in Zustand (or any JS library) is architectural duplication — a state machine inside another state machine.

3. **Direct bindings eliminate sync**: React components call `get_state()` to read and `dispatch(command)` to write. No intermediate JS state to synchronize. No stale data. No competing authorities.

4. **Tiptap is a view, not a state authority**: Tiptap's internal ProseMirror state represents editor-local formatting and selection, not domain data. Commands flow from WASM → React → Tiptap. The WASM core owns the truth.

5. **UI state ≠ domain state**: Panel visibility, scroll position, and active tab are UI-local concerns. React context or component-local state handles these trivially without a state library.

## Consequences

- **No JS state library in dependencies**: Reduces bundle size, eliminates sync complexity.
- **WASM binding API**: `get_state()` and `dispatch(command)` must be performance-optimized — these are called frequently by React components.
- **Performance concern**: `get_state()` serializes state across the WASM boundary. For large documents (10,000+ blocks), this must be benchmarked.
- **Optimistic updates**: If dispatch latency is noticeable, a command queue pattern with optimistic UI updates may be needed (pending benchmark).
- **Tiptap integration**: Tiptap's `editor.commands` must be wrapped to dispatch through WASM first, then update the editor view.
