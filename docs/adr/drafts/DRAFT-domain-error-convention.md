# DRAFT ADR: DomainError Convention — Single String Field

- **Status:** draft (auto-generated by auto-grill-loop)

- **Source:** Q011-P2 (Pass 2)

## Context

Q002-P1 mandated changing all domain `parse_str` functions from `Option<T>` to `Result<T, DomainError>`. This required a new `DomainError` variant for parse failures. The Proxy proposed `ParseError { type_name: String, input: String }` — a two-field variant. However, all existing DomainError variants use a single String field: `InvalidData(String)`, `InvalidConfiguration(String)`, `NotFound(String)`.

The question: should ParseError follow the existing single-field convention or introduce a new multi-field pattern?

## Decision

**ParseError(String)** — single String field following existing DomainError conventions.

Structured information (type_name, input) is embedded in the message string:
```rust
DomainError::ParseError(format!("Failed to parse BlockId from '{}'", input))
```

All DomainError variants follow the same single-String-field convention. No new field shapes introduced.

## Why

1. **Consistency:** DomainError has an established convention across all existing variants. Introducing a two-field variant for ParseError would create inconsistency with no compensating value.

2. **Metadata-as-string anti-pattern:** Splitting structured info into separate String fields (type_name, input) creates fields that are individually meaningless — neither is independently queryable as a typed value. The message string already carries the same information in a single coherent format.

3. **No typed queryability gained:** type_name and input are both Strings — not domain types. They add match-arm surface area without enabling typed error handling.

4. **Pattern stability:** A single convention across all variants makes DomainError predictable, matchable, and maintainable. Future variants follow the same shape without design churn.

## Consequences

- **CONTEXT.md:** Document DomainError convention — single String field for all variants.
- **API:** `DomainError::ParseError(String)` added. All parse_str callers construct ParseError with domain-type name in message.
- **Tests:** ParseError construction and Display impl tested for message format consistency.
- **Observability:** Parse errors flow through DomainError channel with structured, human-readable messages.
- **No persistence impact.**
