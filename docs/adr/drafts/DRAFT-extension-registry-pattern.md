# DRAFT ADR: ExtensionRegistry — Type-Erased DI Container

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-13
- **Source:** Q010-P2 (Pass 2)

## Context

Q001-P1 mandated individual Axum `Extension<T>` per use case, eliminating AppState and AppServices. The initial implementation approach proposed field-per-extension AppState — AppState holds each `Arc<dyn Trait>` as a separate field. This is structurally identical to the god object Q001-P1 eliminated: same shape with Arc wrappers instead of raw fields.

Adding a new Extension requires modifying AppState — a modification hot-spot. The true test from Q001-P1: can you add a new dependency without touching the container? Field-per-extension AppState fails this test.

## Decision

**ExtensionRegistry pattern.** AppState holds ONE field: `extensions: ExtensionRegistry`. The registry is a type-erased container:

```rust
pub struct ExtensionRegistry {
    extensions: HashMap<TypeId, Box<dyn Any + Send + Sync>>,
}

impl ExtensionRegistry {
    pub fn register<T: Send + Sync + 'static>(&mut self, ext: T) { ... }
    pub fn get<T: Send + Sync + 'static>(&self) -> Option<&T> { ... }
}

impl FromRef<ExtensionRegistry> for Arc<dyn BlockUseCases> { ... }
```

- `FromRef` implemented on `ExtensionRegistry`, NOT on AppState directly
- Each use case trait gets its own `FromRef` impl against ExtensionRegistry
- Handlers extract via `Extension(block_uc): Extension<Arc<dyn BlockUseCases>>`
- Adding a new Extension NEVER touches AppState

## Why

1. **Zero god objects:** AppState holds one registry. The registry knows about types via TypeId — AppState never references individual extension types.

2. **Zero modification hot-spots:** Registering a new Extension is a single call to `registry.register()` at bootstrap. AppState is never modified.

3. **Type safety preserved:** `FromRef` impls are per-type. Missing extension = runtime diagnostic panic with clear error message (not silent None propagation).

4. **Testability:** `ExtensionRegistry` is independently testable. Mock extensions register via the same `register()` call — no special constructor.

## Consequences

- **CONTEXT.md:** Document ExtensionRegistry as standard DI container pattern.
- **API:** ExtensionRegistry public API: `register()`, `get()`, `FromRef` impls.
- **Tests:** ExtensionRegistry unit tests: register, get, type-safety, missing-type diagnostics.
- **Observability:** Registered extension count, missing extension diagnostics with type name in error message.
- **No persistence impact.**
