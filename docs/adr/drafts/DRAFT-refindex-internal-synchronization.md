# DRAFT ADR: RefIndex Internal Synchronization

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-13
- **Source:** Q003-P1 (Pass 1)

## Context

`RefService` in `quilt-application` currently exposes its `RwLock` externally, requiring every call site to acquire and release the lock manually. This creates several architectural problems:

1. **Layer violation** — synchronization (a data-consistency concern) lives in the orchestrator (RefService) instead of the data structure (RefIndex) that owns the data being synchronized.

2. **Async safety bug** — the current code holds the lock across `sync_refs().await`, violating Rust's "never hold a synchronous lock across `.await`" rule. A `parking_lot::RwLock` held across an await point blocks the tokio worker thread.

3. **Lock type unspecified** — callers don't know whether to expect `parking_lot`, `std::sync`, or `tokio::sync` lock semantics. Using the wrong type (e.g., `parking_lot` with tokio) silently starves the async runtime.

4. **WASM confusion** — the Proxy claimed WASM compatibility as a justification, but `quilt-application` is a server-side crate. Only `quilt-core` targets WebAssembly. The WASM argument is factually incorrect.

`RefIndex` is the canonical cache of reference relationships between blocks. As a cache, its consistency semantics are eventual — brief inconsistency between index and persistence is acceptable because the source of truth is the SQLite persistence layer.

## Decision

**Push synchronization into RefIndex (the data structure), not RefService (the orchestrator).**

### Architecture

```
RefService (orchestrator)          RefIndex (data structure)
─────────────────────────          ──────────────────────────
&self methods                      tokio::sync::RwLock<RefIndexInner>
Calls RefIndex methods             Owns internal synchronization
No lock awareness                  Lock released before any await
Business logic only                Pure data access
```

### Implementation sketch

```rust
// RefIndex owns the lock
pub struct RefIndex {
    inner: tokio::sync::RwLock<RefIndexInner>,
}

struct RefIndexInner {
    forward: HashMap<BlockId, Vec<Ref>>,
    backlinks: HashMap<BlockId, Vec<Ref>>,
    // ... other indices
}

impl RefIndex {
    /// Resolve references — read-only, lock released before caller can do I/O
    pub async fn resolve(&self, target: &BlockId) -> Vec<Ref> {
        let guard = self.inner.read().await;
        guard.forward.get(target).cloned().unwrap_or_default()
    } // lock released here

    /// Rebuild from persistence — I/O outside lock, lock only for mutation
    pub async fn rebuild_from(&self, repo: &dyn RefRepo) -> Result<(), Error> {
        let refs = repo.load_all().await; // I/O OUTSIDE lock
        let mut guard = self.inner.write().await;
        guard.replace_all(refs); // lock held only for mutation
    } // lock released
}

// RefService orchestrates — no lock awareness
impl RefService {
    pub async fn resolve_refs(&self, block_id: &BlockId) -> Result<Vec<Ref>, Error> {
        let cached = self.index.resolve(block_id).await;
        if cached.is_empty() && self.needs_sync(block_id).await? {
            self.sync_refs().await?; // lock released before this I/O
            return self.index.resolve(block_id).await;
        }
        Ok(cached)
    }
}
```

### Lock type: `tokio::sync::RwLock`

- Async-aware — integrates with tokio runtime
- Won't block worker threads when held across yield points
- Standard choice for async Rust applications

### DashMap evaluated and rejected

DashMap was evaluated as a lock-free alternative that could eliminate the `RwLock` question entirely. It was rejected because:

- RefIndex may need cross-map atomicity (e.g., adding a reference updates both `forward` and `backlinks` collections atomically)
- DashMap provides per-shard locking, not cross-collection atomicity
- Wrapping all inner state in a single `RwLock` is simpler and provides guaranteed consistency for multi-collection operations

### Accepted trade-off: cache inconsistency window

During `rebuild_from()`, there is a brief window where RefIndex is being updated while persistence has already been read. This is acceptable because:

- RefIndex is a **cache**, not the source of truth
- Persistence (SQLite) is always authoritative
- Read operations during rebuild get either the old index state or the new one — never a partial/inconsistent state

## Why

1. **Layer correctness** — data structures own internal consistency (locks), services own business logic (orchestration). Placing the lock in RefService violates Clean Architecture's dependency direction.

2. **Async safety** — RefIndex methods release the lock before any `.await` on I/O. RefService can't accidentally hold the lock during I/O because it never acquires a lock directly. This eliminates the "lock across await" bug structurally.

3. **Specified lock type** — `tokio::sync::RwLock` is the correct choice for a tokio-based server. The type is explicit in the code and verified at compile time.

4. **Alternative evaluated** — DashMap was considered and rejected with documented reasoning, not ignored.

5. **Testability** — RefIndex can be unit-tested for concurrent access patterns independently of RefService orchestration logic.

## Consequences

### Positive

- Clear separation: RefIndex owns synchronization, RefService owns orchestration
- "Lock across await" bug eliminated structurally — impossible by design
- Lock type is specified and tokio-compatible
- RefIndex is independently testable for concurrent access
- Service code is simpler — no lock management in business logic

### Negative

- RefIndex public API becomes async (`&self` → `&self` with `.await`)
- All RefIndex callers must be in async context
- Brief cache inconsistency window during rebuild (documented as acceptable)
- DashMap simplicity is lost in favor of explicit `RwLock` (trade-off for cross-map atomicity)

### Migration

- Refactor `RefIndex` to own `tokio::sync::RwLock<RefIndexInner>`
- Remove lock acquisition/release from all `RefService` methods
- Remove lock exposure from `RefService` public API
- Add concurrent access tests for `RefIndex`
- Document cache consistency semantics in CONTEXT.md
