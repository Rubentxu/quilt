# DRAFT ADR: AppState Elimination — Gradual Rollout Strategy

## Context

Q001-P1 mandated eliminating AppState and AppServices in favor of individual Axum `Extension<T>` registrations per use case, plus an `ExtensionRegistry` pattern (Q010-P2) to avoid creating a new god object. The presentation layer has 73 `pool.clone()` sites and 55 `SqliteBlockRepository::new()` sites — 128+ handler sites total.

Atomic migration of all 128+ sites in a single commit is architecturally reckless: one mistake and the entire system regresses. An incremental strategy is needed that:
- Groups migration work into independently reviewable and revertible units
- Avoids feature-flag complexity (dual-wielding across the system)
- Establishes the new pattern before widespread adoption

## Decision

**Gradual rollout: 6 handler-group PRs. No feature flags. Groups follow domain boundaries.**

| PR | Scope | Description |
|----|-------|-------------|
| PR 0 | Foundation | ExtensionRegistry + Extension<T> pattern infrastructure. Zero handler changes. Establishes `ExtensionRegistry::register()`, `get()`, and `FromRef`. |
| PR 1 | Blocks | Block handlers group migrated to Extension<T>. Uses ExtensionRegistry foundation. |
| PR 2 | Pages | Page handlers group migrated. |
| PR 3 | Search | Search handlers group migrated. |
| PR 4 | Navigation + Infra | NavigationChannel + infrastructure handlers migrated. |
| PR 5 | MCP Tools | MCP tool handlers migrated (last — depends on all registry entries). |
| PR 6 | Cleanup | Delete AppState, AppServices, bootstrap.rs legacy code. Zero AppState references remain. |

Each PR is independently testable and revertible. Dual-wielding is contained within PR scope — AppState and Extensions coexist temporarily for one handler group at a time, never across the entire system.

## Why

1. **Safety:** 6 small, reviewable PRs with clear failure boundaries. Any PR can be reverted independently without affecting other handler groups.

2. **No feature flags:** Q001-P1 explicitly rejected split-brain architectures. Feature flags would create conditional complexity in every handler. Domain-boundary grouping avoids this — within each PR, the handler group uses Extensions consistently.

3. **PR 0 as pattern anchor:** Establishing ExtensionRegistry + Extension<T> before any migration means subsequent PRs follow a proven, tested pattern. Reviewers learn the pattern once in PR 0.

4. **Domain boundaries minimize coupling:** Handlers in the same domain group share repository and use-case dependencies. Cross-group coupling is minimal — each PR's scope is self-contained.

5. **Cleanup as final PR:** PR 6 deletes legacy code when zero references remain — verifiable by grep. No dead code accumulates during migration.

## Consequences

- **CONTEXT.md:** Document rollout strategy, PR grouping, and migration verification metric.
- **API:** Each PR changes handler Extension signatures. PR 0 introduces ExtensionRegistry public API.
- **Tests:** Each PR independently testable. Integration tests validate full migration after PR 6.
- **Observability:** PR-level verification of Extension adoption. Zero AppState references metric after PR 6.
- **No persistence impact.**
