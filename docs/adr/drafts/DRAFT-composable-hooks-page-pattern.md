# DRAFT ADR: Composable Hooks Pattern for Page Components

## Context

PageView.tsx, the primary page rendering component in Quilt's React frontend, had grown into a God Component. The question arose: should it become a pure presentation component with a monolithic `usePageController` hook?

Investigation revealed that PageView.tsx already used 3 composable hooks successfully (`usePageBlocks`, `useBlockSync`, `usePageUndoManager`). The codebase had already established composable hooks as the pattern. Additionally, ~310 lines of deprecated comment handlers were identified as dead code eligible for deletion. Several cross-cutting concerns (drag-and-drop, selection, comments, annotations, focus management, zoom, templates, kanban, favorites) were either scattered across PageView or absent entirely.

## Decision

**PageView becomes an orchestrator (~300-400 lines) composing ~10 focused hooks.** Each hook owns exactly one concern. The monolithic `usePageController` approach is explicitly rejected as a regression from the already-proven composable hooks pattern.

Hooks to preserve:
- `usePageBlocks`
- `useBlockSync`
- `usePageUndoManager`

New hooks to create:
- `useBlockOperations`
- `useBlockDragAndDrop`
- `useBlockSelection`
- `useComments`
- `useAnnotations`
- `useFocusManagement`
- `useBlockZoom`
- `useTemplateIndex`
- `useKanbanView`
- `usePageFavorite`

Sub-components to extract:
- `SortableBlockRowFlat` тАФ row rendering
- `BlockErrorBoundary` тАФ error isolation
- `JournalDateHeader` тАФ date display
- `EmptyState` тАФ empty page state
- `PageViewError` тАФ error state

Supporting changes:
- DELETE deprecated comment handlers (~310 lines) тАФ dead code is removed, not migrated
- Extend `BlockContext` for props drilling
- Each hook independently testable; each sub-component renders in isolation

## Why

The Proxy proposed a monolithic `usePageController` without checking what already existed. PageView.tsx already used 3 composable hooks successfully. Replacing them with a single controller would be a clear regression:

1. **Fewer, larger abstractions are harder to test** тАФ a monolithic hook requires complex mock setup; focused hooks test in isolation
2. **Fewer, larger abstractions are harder to reason about** тАФ each focused hook has a single responsibility
3. **Fewer, larger abstractions are harder to compose** тАФ focused hooks can be reused across components
4. **Deletion over migration** тАФ dead code (~310 lines of comment handlers) should be removed, not carried forward into new abstractions

The composable hooks pattern scales: each hook owns one concern, is independently testable, and composes cleanly in the orchestrator.

## Consequences

- PageView.tsx refactored from God Component to orchestrator + hooks + sub-components
- ~310 lines of dead code removed
- ~10 new focused hooks created
- 5 sub-components extracted for rendering, error, empty, and error states
- BlockContext extended for props drilling
- Each hook becomes an independent unit of testing, monitoring, and reasoning
- Pattern can be replicated across other page components (AppShell, Sidebar)
