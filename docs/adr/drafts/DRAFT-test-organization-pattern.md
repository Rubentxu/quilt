# DRAFT ADR: Test Organization Pattern — Per-File #[cfg(test)] + tests/common.rs

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-13
- **Source:** Q006-P1 (Pass 1)

## Context

`quilt-infrastructure` has a 926-line monolithic `#[cfg(test)]` module in `repositories.rs` covering tests for 9+ SQLite repository implementations. This creates discoverability and maintenance friction:

- Developers must navigate a single large file to find tests for a specific repo
- Merge conflicts are more likely when multiple people touch different repos' tests
- Shared test helpers (fixtures, seeded data, assertion utilities) have no canonical home
- The pattern is inconsistent with Rust ecosystem conventions

The Rust ecosystem standard is per-file `#[cfg(test)] mod tests` at the bottom of each implementation file. Quilt should adopt this pattern workspace-wide.

## Decision

**Per-file `#[cfg(test)] mod tests` for all Rust crates. Shared test helpers live in `tests/common.rs` within each crate.**

For `quilt-infrastructure`:
- Each repo file (`block_repo.rs`, `page_repo.rs`, `ref_repo.rs`, etc.) gets its own `#[cfg(test)] mod tests` at the bottom
- Shared helpers (setup_db, fixtures, assertions) extracted to `tests/common.rs` — `cfg(test)`-only
- `repositories.rs` becomes a ~38-line re-export hub with zero test code
- No visibility changes needed — all test targets already accessible within the crate

For all other crates:
- Same pattern applies: per-file `#[cfg(test)]` modules
- Each crate may have its own `tests/common.rs` for shared test utilities
- `quilt-test-helpers` continues providing in-memory doubles only (cross-crate test support)

## Why

1. **Rust ecosystem standard**: Per-file `#[cfg(test)]` is the dominant convention in the Rust community
2. **Discoverability**: Developer opens `block_repo.rs`, scrolls to the bottom, finds the tests immediately
3. **Maintenance**: Per-repo test scope reduces merge conflicts and cognitive load
4. **Shared helpers get a home**: `tests/common.rs` eliminates informal sharing and duplicated fixtures
5. **No architectural impact**: Pure file organization — zero visibility changes, zero test logic changes
6. **Mechanical verification**: Test count before/after must be identical (zero regression)

The Skeptic's concerns are execution risks, not design risks:
- `tests/common.rs` god-module risk → mitigated by organizing into sub-modules by domain (fixtures, assertions, setup)
- Migration risk → mitigated by mechanical test count comparison and diff review

## Consequences

- **Tests:** Structural change only — zero test logic changes. Each repo's tests are colocated with its implementation.
- **ADR:** This document establishes the pattern workspace-wide.
- **CONTEXT.md:** Updated to document per-file `#[cfg(test)]` as the standard pattern.

No impact on API, persistence, security, or observability.
