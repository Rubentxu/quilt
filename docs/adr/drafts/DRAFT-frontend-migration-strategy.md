# DRAFT ADR: Frontend Migration Strategy — Leptos to React

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-05-30
- **Source:** Q004-P1 (Pass 1)

## Context

Quilt is migrating its frontend from Leptos 0.8 to React + TypeScript with a hybrid Rust/WASM boundary (see DRAFT-hybrid-wasm-boundary). The existing Leptos application is functional and must not be broken during migration. The migration involves:

- ~3,692 lines of outliner core staying in Rust/WASM
- ~2,000+ lines of inline parser staying provisionally in Rust/WASM
- ~1,500+ lines of Leptos views to be rewritten in React
- ~300 lines of wasm-bindgen CM6 FFI to be replaced with native JS editors
- Editor engine change: CM6 (primary) → Tiptap (primary outliner) + CM6 (code blocks only) (see DRAFT-editor-engine-selection)

The critical path is WASM-React interop: if calling Rust/WASM outliner functions from React components has unexpected friction, the entire architecture must be re-evaluated. This critical path is currently unproven — no POC exists.

## Decision

**Three-phase parallel-track migration with a mandatory Phase-1 de-risking gate.**

### Phase 1 — De-risking Gate (mandatory, target ~2 weeks)

1. **Extract portable Rust** into shared crate `quilt-core`: outliner core (~14 files), not the ~33 originally estimated (excludes config, build glue, CM6 bindings)
2. **Build WASM-React interop POC**: one real tree operation (e.g., "insert block") called from WASM, result rendered in a React component
3. **Benchmark inline parser**: WASM vs TypeScript for real-time keystroke formatting
4. **Define parity as closed checklist**: explicit, testable items covering all Leptos features (block CRUD, journal navigation, sidebar, properties, references, search, slash commands)
5. **Set concrete Leptos sunset date** (proposed: Q3 2026)

**Gate review**: All Phase 1 items pass → proceed to Phase 2. Any item fails → re-evaluate architecture (fallback options: full TypeScript rewrite of outliner, or alternative WASM boundary).

### Phase 2 — Full React Build (after gate passes)

1. Create `quilt-ui-react` with Vite + React + TypeScript + Tiptap
2. Implement components against closed parity checklist
3. Each sprint: track checklist completion percentage
4. Switchover when checklist = 100%

### Phase 3 — Cleanup (post-switchover)

1. Remove `quilt-ui` Leptos crate
2. Remove Trunk build configuration
3. Update ADRs: 0005 (superseded), 0007 (revised), 0010 (updated)
4. Archive migration artifacts

## Why

1. **Critical path first**: WASM-React interop is the single thing that, if it fails, collapses the entire parallel-track strategy. The Phase 1 gate proves it works (or reveals it doesn't) before months of investment in Phase 2.

2. **Measurable progress**: A closed parity checklist transforms "are we done yet?" from a sentiment to a metric. Without it, "90% done" can persist indefinitely while Leptos accrues changes.

3. **Prevent indefinite dual-maintenance**: A concrete Leptos sunset date forces the decision of when old code is removed. Without it, "temporary coexistence" becomes permanent.

4. **Precise scope**: The Proxy originally claimed ~33 portable files. Actual analysis shows ~14 are truly portable (excluding config, build glue, and CM6 bindings that are replaced, not ported). Precise counting prevents false velocity expectations.

## Consequences

- **ADR-0005 (Leptos CSR)**: Superseded upon switchover. React replaces Leptos.
- **ADR-0007 (CM6)**: Revised — CM6 role changes to code blocks only.
- **ADR-0010 (testing)**: Updated — adds Jest/Vitest/Playwright alongside Rust tests.
- **New ADRs required**: WASM boundary (DRAFT-hybrid-wasm-boundary), editor selection (DRAFT-editor-engine-selection), migration strategy (this ADR).
- **Build pipeline**: Trunk → Vite. Two build tools during Phase 2 coexistence.
- **Team workflow**: Rust + TypeScript development during migration. CI/CD must support both.
- **Rollback**: Leptos remains functional until switchover. Rollback is "don't switch over."
