# DRAFT ADR: Property Editing Surface Architecture

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-03
- **Source:** Q003-P1 (Pass 1)

## Context

Quilt needs a property editing surface. Properties are block-attached structured key-value pairs (ADR-0007). The question is where in the UI this editing surface should live.

Three primary options were considered:
1. **Inline-first hybrid**: Properties rendered inline within block content with 3 visibility tiers (headline pills, expanded, hidden). Block-scoped inline editing. Right sidebar deferred to V2.
2. **Right-sidebar panel**: Properties in a dedicated sidebar panel (Notion-style).
3. **Fixed header section**: Always-visible property header between page title and content area. Block focus populates the same header.

## Decision

**Fixed always-visible property header section below page title.**

- No visibility tiers. Header is always present.
- Properties are NEVER inline within block content.
- Block-focused editing populates the SAME header (no sidebar).
- Mobile: compact sticky header + expandable bottom sheet.

### What was explicitly rejected

- ❌ Property rendering inline within block content — category error (structured metadata inside unstructured prose)
- ❌ Three-tier visibility state machine — CRDT complexity (3^N states across N properties)
- ❌ "4 pinned" magic number — arbitrary cap with no user-model justification
- ❌ Right sidebar for v1 — deferred complexity, two-surface editing model

## Why

1. **Category boundary enforcement**: Properties are structured metadata. Block content is unstructured prose. Mixing them creates a category error that ripples through sync (CRDT), search (need to parse prose), and AI surface (need content extraction).

2. **CRDT simplicity**: Fixed header = exact 2 states (empty/populated). Three-tier visibility = 3 visibility states to sync per property. The header model eliminates sync conflicts from visibility toggling.

3. **No magic numbers**: Every property has equal structural importance. User-controlled ordering (reorder, not pin-4) replaces arbitrary caps.

4. **AI-first architecture**: Structured metadata in a known, fixed position is directly machine-queryable. Inline properties would require content parsing — a regression in AI accessibility.

5. **Mobile consistency**: Compact sticky header + bottom sheet is a simpler two-surface model than inline condensing, which would require responsive reflow of content-embedded properties.

6. **Single editing surface**: Block focus populates the same header — users never context-switch between surfaces for metadata editing vs content editing.

## Consequences

### Positive
- Single source of truth for property display
- Minimal CRDT state transitions
- AI agents can query properties by position, not by parsing content
- No magic numbers or arbitrary caps
- Mobile model is consistent with desktop model (sticky header + sheet)

### Negative
- Less "inline" feel — properties are separated from content visually
- Header area consumes vertical space even when empty
- Users accustomed to Notion-style inline properties will need adjustment
- Block-focus ↔ header population interaction model needs careful design

### Neutral (design decisions to resolve)
- Exact interaction model for block-focus → header population
- Property reordering persistence
- Auto-population behavior on page load
- Mobile bottom sheet content strategy
