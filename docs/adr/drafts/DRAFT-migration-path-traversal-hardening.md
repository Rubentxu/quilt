# DRAFT ADR: Migration Endpoint — Path Traversal Hardening Policy

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-13
- **Source:** Q004 (Pass 1)

## Context

The migration endpoint `POST /api/v1/migration/md` accepts a user-supplied directory path and imports all Markdown files found there. The current validation is minimal:

```rust
// migration.rs:143-155
if !path.exists() {
    return Err(AppError::BadRequest(...));
}
if !path.is_dir() {
    return Err(AppError::BadRequest(...));
}
```

`import_directory` then reads entries directly with `std::fs::read_dir(dir_path)` without canonicalization or prefix checks. An attacker could supply a path like `/etc/../../../var/data/imports` that resolves to an unexpected directory after following symlinks or `..` traversal.

The server runs with the permissions of the process user — a path traversal could read files outside the intended import directory.

## Decision

**Two-phase hardening: minimum viable now, full hardening deferred.**

### Phase 1 — Minimum viable (implement immediately)

1. **Canonicalize** the user-supplied path with `std::fs::canonicalize` before any I/O
2. **Prefix check**: verify the canonicalized path starts with a configured allowed import root (default: process working directory)
3. **Reject** non-canonical or out-of-bounds paths with `400 Bad Request`

```rust
let allowed_root = std::env::current_dir()?;
let canonical = std::fs::canonicalize(&path)?;
if !canonical.starts_with(&allowed_root) {
    return Err(AppError::BadRequest("Path outside allowed import root"));
}
```

### Phase 2 — Full hardening (deferred to follow-up)

| Concern | Mitigation | Priority |
|---------|-----------|----------|
| TOCTOU (time-of-check/time-of-use) | Open directory handle at check time, use `openat`-style operations | Medium |
| Symlink attacks | `canonicalize` resolves symlinks (Phase 1 covers this) | Done |
| File count DoS | Max files per import (e.g., 1000) | Low |
| File size DoS | Max size per file (e.g., 10 MB) | Low |
| Recursive symlink loops | `canonicalize` fails on loops with `ELOOP` (Phase 1 covers this) | Done |
| Allowed root configuration | Environment variable `QUILT_IMPORT_ROOT` | Low |

## Why

1. **Phase 1 is sufficient for the current threat model**: Quilt is a single-user desktop application. The migration endpoint is authenticated (Bearer token). The primary risk is accidental misconfiguration, not malicious multi-tenant attacks.

2. **`canonicalize` resolves symlinks and `..` components atomically**: It follows symlinks to their ultimate target and normalizes the path. An attacker cannot use `..` or symlinks to escape the allowed root.

3. **Deferred hardening avoids scope creep**: TOCTOU hardening and DoS limits require architectural changes (file handle passing, middleware-level limits). These are real concerns but not blocking for initial deployment.

4. **Prefix check is simple and auditable**: Three lines of code with clear semantics. Easy to review, easy to test.

## Consequences

- **CONTEXT.md**: Document the import root constraint. Document that `canonicalize` resolves symlinks.
- **ADR**: This is the ADR. Promotes to numbered ADR after Phase 1 implementation and Phase 2 scoping.
- **API**: Migration endpoint now rejects paths outside the allowed root. Error response changes from generic "path does not exist" to specific "path outside allowed import root".
- **Persistence**: No impact.
- **Tests**: New tests for canonicalization, symlink traversal, `..` escape attempts. Test with temp directories.
- **Security**: Closes path traversal vector. Does not address TOCTOU race (deferred).
- **Observability**: Log rejected path attempts for audit trail.
