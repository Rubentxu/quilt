# DRAFT ADR: Backend Protocol — REST + SSE + Notify

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-05-30
- **Source:** Q005-P1 (Pass 1)

## Context

Quilt's current frontend (Leptos) uses HTTP REST for CRUD operations and a 386-line polling loop for state synchronization. The polling loop periodically queries the backend for changes — a workaround for the absence of push infrastructure, not an architectural choice.

The React frontend migration (see DRAFT-hybrid-wasm-boundary) is an opportunity to replace polling with proper server→client push. The backend stack (Rust + SQLite + Tokio running quilt-server) is unchanged.

The `notify` crate (Rust filesystem watcher: inotify/kqueue/FSEvents) can detect domain changes at the filesystem/database level and trigger push events. SSE (Server-Sent Events) provides browser-native unidirectional server→client streaming via the `EventSource` API. ts-rs generates TypeScript type definitions from Rust structs, and OpenAPI documents the REST API.

## Decision

**REST primary (OpenAPI-documented). Replace 386-line polling loop with `notify` crate → SSE push. ts-rs + OpenAPI for fully typed API bridge.**

| Component | Technology | Role |
|-----------|-----------|------|
| CRUD operations | REST (OpenAPI) | Standard HTTP methods for blocks, pages, journals, properties |
| State sync (server→client) | SSE (notify-triggered) | Push block/page changes, journal updates, reference notifications |
| Type bridge | ts-rs + OpenAPI | Rust struct → TypeScript type generation; REST endpoint documentation |
| Real-time bidirectional | WebSocket | MCP only — bidirectional JSON-RPC (see DRAFT-mcp-client-architecture) |
| Change detection | notify crate | Filesystem/database change events → SSE triggers |

**Delete**: 386-line polling loop in `quilt-ui`.

## Why

1. **Polling is a workaround, not an architecture**: Periodically querying the backend for changes wastes bandwidth, adds latency (up to poll interval), and doesn't scale. SSE provides push with automatic reconnection and browser-native EventSource API — simpler than WebSocket for unidirectional state sync.

2. **notify crate is already available**: Rust's `notify` crate provides cross-platform filesystem watching. Wiring it to SSE domain event handlers adds minimal code compared to maintaining a polling loop.

3. **SSE > WebSocket for state sync**: SSE is unidirectional (server→client), uses standard HTTP, has automatic reconnection, and is natively supported in browsers. WebSocket adds bidirectional handshake complexity that's unnecessary for state change notifications. WebSocket is reserved for MCP where bidirectional JSON-RPC is genuinely needed.

4. **ts-rs + OpenAPI = typed bridge with zero manual sync**: ts-rs generates TypeScript types from Rust structs at build time. OpenAPI documents REST endpoints. Together they ensure frontend and backend types stay synchronized without manual type definition maintenance.

## Consequences

- **Delete**: ~386 lines of polling code removed.
- **New dependencies**: notify crate (Rust, server-side), OpenAPI generation (utoipa or similar), SSE implementation (axum SSE or tokio-stream).
- **SSE event types**: Must be defined — block_created, block_updated, block_deleted, page_renamed, journal_entry_added, reference_added, etc.
- **SSE reconnection**: Strategy needed for missed events during disconnection (event IDs, last-event-ID header, replay/resync mechanism).
- **CORS**: SSE requires proper CORS configuration for EventSource (different from fetch CORS).
- **MCP WebSocket**: Separate port or same port as REST/SSE — architectural decision pending.
