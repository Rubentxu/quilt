# DRAFT ADR: Hybrid WASM Boundary for React Frontend

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-05-30
- **Source:** Q002-P1 (Pass 1)

## Context

Quilt is migrating its frontend from Leptos 0.8 (Rust/WASM) to React + TypeScript. The current `crates/quilt-ui/src/` contains ~76 Rust files (~5,000+ lines) spanning domain logic, inline parsing, editor integration, and Leptos-specific views. A key architectural question: what stays in Rust/WASM and what moves to TypeScript?

The outliner core (~3,692 lines) is pure domain logic — block CRUD, tree traversal, journal navigation, reference resolution, property management — with zero UI framework coupling. It is tested and functional.

The inline parser (~2,000+ lines) handles real-time keystroke formatting (marks, Logseq-compatible syntax). It is algorithmically portable but its performance in WASM vs TypeScript is unmeasured.

The CodeMirror 6 wasm-bindgen layer (~300 lines) wraps a JavaScript library with manual FFI. In a React context, native JS CM6 integration becomes possible, obviating this layer.

The Leptos view layer (~1,500+ lines) is tightly coupled to Leptos signals and must be rewritten regardless.

## Decision

**Hybrid architecture.** The WASM boundary is drawn as follows:

| Component | Placement | Rationale |
|-----------|-----------|-----------|
| Outliner core (~3,692 lines) | **Rust/WASM** | Tested, framework-independent, algorithmic. No benefit from TypeScript rewrite. |
| Inline parser (~2,000+ lines) | **Rust/WASM (provisional)** | Stays in WASM pending benchmark. If WASM overhead is unacceptable for real-time keystroke formatting, move to TypeScript. |
| Editor engine (CM6) | **TypeScript (native JS)** | Obviates ~300 lines of wasm-bindgen. Native JS CM6 is cleaner. Tiptap (ProseMirror) becomes primary outliner editor (see DRAFT-editor-engine-selection). |
| UI shell (views, routing, components) | **React/TypeScript** | Rewrite required. Leptos-specific code is not portable. |
| Query DSL | **To be decided** | Separate decision. Evaluate placement independently. |

**Type sharing**: Use `ts-rs` crate to generate TypeScript type definitions from Rust structs for API contracts and WASM interface types.

## Why

1. **Decomposition over false binary**: The WASM boundary is not one decision but 4+ independent decisions. Collapsing them into "rewrite all" or "keep all" discards nuance and creates unnecessary risk.

2. **Keep proven code**: The outliner core has zero UI framework coupling and existing tests. Rewriting ~3,692 lines of tested domain logic in TypeScript costs 2-4 weeks minimum and introduces regression risk with zero architectural benefit.

3. **Proven pattern**: Rust/WASM core + React UI shell is used in production by Figma, 1Password, and others. ts-rs provides type safety across the boundary.

4. **De-risk the unknown**: The inline parser's performance in WASM for real-time keystroke formatting is unmeasured. Provisional placement with a benchmark gate prevents premature commitment.

## Consequences

- **Two language UI layer**: Rust/WASM for domain logic, TypeScript/React for UI shell. Adds TypeScript to the stack but replaces Leptos views, not a net increase.
- **ts-rs dependency**: New build dependency for type generation.
- **WASM-React interop**: Must be validated via POC (see DRAFT-frontend-migration-strategy). If interop has unexpected friction, architecture must be re-evaluated.
- **ADR-0005 (Leptos CSR)**: Superseded. React replaces Leptos; WASM boundary redefined.
- **ADR-0001 (MCP-first)**: Unchanged. MCP is transport-agnostic.
