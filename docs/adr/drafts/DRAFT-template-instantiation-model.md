# DRAFT ADR: Template Instantiation Model — Copy-on-Instantiate + Reapply

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-03
- **Source:** Q002-P1 (Pass 1)

## Context

Quilt's template system, defined in ADR-0007, uses a block-level `template::` property model. Pages can be created from templates via `POST /api/v1/pages/from-template`, which clones the template's block tree at creation time — implementing copy-on-instantiate by default.

ADR-0007 explicitly defers template versioning and snapshot behavior to V2: "Versionado de templates (snapshot inmutable al aplicar)."

The auto-grill explored whether, when a page is created from a template, property schemas should maintain a live link to the source template (Notion model: editing the template updates all derived pages) or should be copied at instantiation time (current behavior).

## Decision

**Copy-on-instantiate** is the V1 behavior. When a page is created from a template, the block tree and properties are cloned at that moment and become independent. No live link to the source template is maintained.

To support template evolution, an explicit **`POST /api/v1/pages/:name/reapply-template`** endpoint allows users to pull updates from a template into an existing page as a deliberate action.

Live schema links (automatic propagation of template changes to derived pages) are deferred to V2.

### V1 Concrete Behavior

1. **`POST /api/v1/pages/from-template`**: Clones the template's block tree. Variable substitution applies. The resulting page has no runtime dependency on the template.
2. **`POST /api/v1/pages/:name/reapply-template`**: Explicit user action. Re-applies the template's current block structure and properties to the target page. Semantics (merge vs overwrite) to be defined in implementation.
3. **No `source_template` field** on pages unless needed for `reapply-template` lookup.
4. **No orphan lifecycle**: Deleting a template page has no effect on pages previously created from it.
5. **No `schema_status` field**: No tracking of whether a page is "stale" relative to its template.

## Why

1. **ADR-0007 alignment**: The settled architecture is block-level `template::`. A page-level live schema link would create a second, incompatible template model coexisting in the same codebase — architectural schizophrenia.
2. **Existing code supports it**: `POST /api/v1/pages/from-template` already clones at instantiation time. The copy-on-instantiate behavior is implemented, not theoretical.
3. **ADR-0007 deferral respected**: Versioning/snapshots are explicitly deferred to V2. Live schema links are a V2+ concern.
4. **Explicit beats implicit**: `reapply-template` as a deliberate user action is simpler, safer, and more predictable than an invisible live link that silently changes pages when templates are edited.
5. **V1 simplicity**: The Proxy's live-schema-link proposal required 9 rules (additive evolution, orphan-on-removal, explicit retyping, user overrides, etc.). An explicit reapply endpoint reduces this to 1 user action.

## Consequences

### Positives

- Simple, predictable behavior: pages don't change unless the user takes action
- No new database columns or migration needed
- No dual template model (block-level `template::` + page-level live schema)
- Existing `from-template` behavior preserved, not replaced
- `reapply-template` gives users control over when to pull template updates

### Negatives

- Templates that evolve leave old pages with stale structure (mitigated by `reapply-template`)
- No automatic propagation of template fixes or improvements to existing pages
- User must remember to reapply templates when they want updates

### Trade-offs accepted

- **Explicit over automatic**: V1 prioritizes user control and simplicity over convenience. Automatic propagation is a V2 feature.
- **No version tracking**: Pages don't know which template version they were created from. This simplifies V1 at the cost of making future diff/reapply logic harder.
