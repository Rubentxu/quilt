# DRAFT ADR: Query Compiler Abstraction and DSL Refinement

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-03
- **Source:** Q004-P1 (Pass 1)

## Context

ADR-0004 established the S-expression DSL as the canonical query representation shared between UI and MCP, with the MCP exposing a superset of operators. However, the current implementation has three critical issues:

1. **No QueryCompiler abstraction**: SQL generation for DSL queries is scattered across three methods in `sqlite_block_repo.rs` with inline version-check conditionals.
2. **Panics in query paths**: `build_where` and related methods contain `panic!()` calls that reach API callers — a malformed query can crash the server.
3. **Underspecified PropertyOp**: The `PropertyOp` type in the AST lacks a clear specification of operators, their semantics, and backward-compatibility guarantees.

Additionally, ADR-0004 assumes users interact with the DSL through free-text, but non-technical users have no visual affordance for constructing property-filter queries. A v1 stopgap is needed before a full visual query builder in V2.

## Decision

1. **DSL-first confirmed** — ADR-0004 stands. S-expression DSL remains the canonical query representation.
2. **Add `QueryCompiler` abstraction** — A single trait or struct that encapsulates all DSL-to-SQL compilation. All code paths (search, property query, MCP query) call the compiler instead of generating SQL inline.
3. **Remove panics from query paths** — `build_where` and all SQL-generation methods return `Result<T, QueryError>`. API callers receive structured errors, never crashes.
4. **Backward-compatible `PropertyOp`** — Define as a closed enum:
   - `Eq`, `Neq`, `Gt`, `Lt`, `Contains`, `Exists`, `Missing`
   - Array `Contains` checks element membership
   - String `Contains` checks substring
   - Existing DSL expressions remain parseable
5. **Frontend filter-chips stopgap** — Implement visual filter-chips (Gmail/Notion/Linear pattern) that serialize to DSL AST. Non-technical users build queries via chip composition; the backend receives structured DSL expressions, not free text.
6. **AST consolidation** — Merge overlapping types between `quilt-query` and query execution: `QueryNode`, `Condition`, `PropertyCondition`.

## Why

1. **ADR-0004 alignment**: DSL-first is settled architecture. No need to revisit.
2. **Crash safety**: Panics in query paths are a security/reliability regression for MCP agents constructing queries programmatically. Structured errors enable graceful failure.
3. **Maintainability**: Three scattered SQL methods with inline version checks create a maintainability time bomb. A single `QueryCompiler` abstraction eliminates the boundary leak.
4. **UX pragmatism**: A full visual query builder is too expensive for v1. Filter-chips are a proven lightweight pattern that bridges the gap — they serialize to DSL AST, preserving the canonical representation.
5. **Backward compatibility**: PropertyOp changes must not break existing queries or stored DSL expressions.

## Consequences

- `quilt-query` crate gains a `QueryCompiler` trait and a `QueryError` type
- All SQL-generation code moves from infrastructure layer into the compiler
- `PropertyOp` becomes a closed enum with defined semantics
- Frontend implements filter-chips component that serializes to DSL AST
- ADR-0004 should be updated to reflect the QueryCompiler abstraction and filter-chips stopgap
- Existing DSL queries must be tested for backward compatibility with the new PropertyOp enum
