# DRAFT ADR: Domain parse_str Returns Result<T, DomainError>, Not Option<T>

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-13
- **Source:** Q002-P1 (Pass 1)

## Context

Quilt's domain layer (`quilt-domain`) defines `parse_str` functions for constructing domain types from strings (e.g., `BlockId::parse_str`, `Uuid::parse_str`). These functions currently return `Option<T>` — `Some(value)` on success, `None` on failure.

This interface forces every call site outside the domain to manually convert `None` to an error:

```rust
let id = Uuid::parse_str(&input).ok_or_else(|| AppError::InvalidUuid)?;
```

Two problems emerge:
1. **Silent-drop bugs**: If a call site forgets the `.ok_or_else()`, the parse failure is silently ignored. `blocks.rs:240` exhibits this exact bug — `parent_uuid` parse failure produces `None`, which propagates as "no parent" instead of "invalid parent UUID".
2. **Inconsistent error messages**: With 15+ call sites across REST and MCP handlers, each independently chooses its error message. REST says "Invalid block ID", MCP says "Failed to parse block ID".

The proposed fix — creating a shared `InputParser` module or per-adapter wrapper traits — treats the symptom (code duplication) rather than the root cause (wrong return type). `Option<T>` communicates optionality ("this might not exist"), but parsing is fallible ("this string is valid or invalid"). The semantics mismatch is the root cause.

## Decision

**Change all `parse_str` functions in `quilt-domain` to return `Result<T, DomainError>` instead of `Option<T>`.**

### Before

```rust
// quilt-domain
impl Uuid {
    pub fn parse_str(s: &str) -> Option<Self> { ... }
}

// Every call site (REST handler, MCP handler, etc.)
let id = Uuid::parse_str(&input).ok_or_else(|| AppError::InvalidUuid)?;
```

### After

```rust
// quilt-domain
impl Uuid {
    pub fn parse_str(s: &str) -> Result<Self, DomainError> { ... }
}

// Every call site
let id = Uuid::parse_str(&input)?;
// DomainError propagates via ?. Each adapter maps it ONCE via From impl.
```

### What does NOT get created

- No shared `InputParser` module
- No `Parseable` trait in the application layer
- No adapter wrapper traits

Each adapter (REST, MCP) creates a single `parse_param::<T>()` inline helper that calls `T::parse_str(s)?` and lets the `?` operator propagate `DomainError` to the adapter's error conversion layer.

### Error mapping

```rust
// In REST adapter (quilt-server)
impl From<DomainError> for ApiError {
    fn from(e: DomainError) -> Self {
        match e {
            DomainError::ParseError(msg) => ApiError::BadRequest(msg),
            // ... other variants
        }
    }
}

// In MCP adapter (quilt-mcp)
impl From<DomainError> for McpError {
    fn from(e: DomainError) -> Self {
        match e {
            DomainError::ParseError(msg) => McpError::InvalidParams(msg),
            // ... other variants
        }
    }
}
```

## Why

1. **Semantic correctness**: Parsing is fallible, not optional. `Result<T, E>` is the Rust idiom for fallible operations. `Option<T>` is for optionality. The domain should express the correct semantics.
2. **Type-system enforcement**: `Result<T, E>` forces every call site to handle the error path. The `?` operator propagates errors automatically. Silent drops become impossible — the compiler rejects code that ignores a `Result`.
3. **Error message consistency**: All parse failures flow through `DomainError`, which carries a single, domain-authoritative error message. Adapters map to protocol-specific errors in one place (`From` impl), not at every call site.
4. **No organizational overhead**: No shared module, no trait, no wrapper abstraction. Each adapter writes `parse_param::<T>(s: &str) -> Result<T, ApiError>` as a one-liner using the `?` operator and `From` conversion.
5. **Future-adapter proof**: A new adapter (WebSocket, CLI, gRPC) only needs to implement `From<DomainError>` for its error type. No wrapper pattern to remember, no shared module to import.

## Consequences

### Positive

- Silent-drop bug class eliminated at the type level (compiler-enforced)
- 15+ inconsistent error strings collapsed into `DomainError` variants + per-adapter `From` impl
- No new abstraction layer (no InputParser module, no Parseable trait)
- All call sites simplified: `.ok_or_else(|| ...)` → `?`
- New adapters need only implement `From<DomainError>` — zero boilerplate per parse site

### Negative

- `DomainError` enum must gain a `ParseError` variant (or per-type variants)
- All existing `parse_str` call sites must be updated (signature change)
- `DomainError` must implement `std::error::Error` + relevant conversion traits
- WASM boundary: `DomainError` must be serializable if parse_str is called from WASM

### Migration

Single commit:
1. Add `ParseError(String)` variant to `DomainError`
2. Change all `parse_str` signatures from `Option<T>` to `Result<T, DomainError>`
3. Remove `.ok_or_else(|| ...)` from all call sites, relying on `?` propagation
4. Add `From<DomainError> for ApiError` and `From<DomainError> for McpError`
5. Verify no call site silently ignores the Result
