# DRAFT ADR: AppState Deletion — Axum Extension Per Use Case

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-13
- **Source:** Q001-P1 (Pass 1)

## Context

`quilt-server` currently uses a monolithic `AppState` struct as a shared context object for all HTTP handlers. This struct bundles infrastructure concerns (pool, navigation_tx, search_index) and application services (block_repo, page_repo) into a single object. The codebase has 73 `pool.clone()` sites and 55 direct `SqliteBlockRepository::new()` calls in the presentation layer — both violations of Clean Architecture (dependencies should point inward, presentation should depend on abstractions).

An existing `AppServices` struct in `bootstrap.rs` already aggregates 12+ `Arc<dyn Trait>` fields as a proposed replacement. However, this replicates the same God Object anti-pattern — an aggregator of traits is structurally identical to a struct with fields when it concentrates too many dependencies.

Axum provides `Extension<T>` for type-based dependency injection. Each extension is registered independently and extracted by type in handler signatures. This pattern eliminates the need for any shared context object.

## Decision

**Eliminate both AppState and AppServices. Register each use case as an individual Axum Extension.**

### The Pattern

```rust
// Registration in server setup
Router::new()
    .layer(Extension(block_use_cases as Arc<dyn BlockUseCases>))
    .layer(Extension(page_use_cases as Arc<dyn PageUseCases>))
    .layer(Extension(search_use_cases as Arc<dyn SearchUseCases>))
    .layer(Extension(navigation_tx))  // infrastructure as dedicated Extension
    .layer(Extension(search_index))   // infrastructure as dedicated Extension

// Extraction in handler — only what's needed
async fn get_block(
    Extension(block_uc): Extension<Arc<dyn BlockUseCases>>,
    Path(id): Path<BlockId>,
) -> Response { ... }

async fn search_blocks(
    Extension(search_uc): Extension<Arc<dyn SearchUseCases>>,
    Extension(search_index): Extension<SearchIndex>,
    Query(params): Query<SearchParams>,
) -> Response { ... }
```

### What gets deleted

- `quilt-server/src/state.rs` — AppState struct
- `bootstrap.rs` — AppServices aggregator struct
- All `Extension<AppState>` extractions in handlers
- All `pool.clone()` calls in presentation layer (73 sites)
- All direct `SqliteBlockRepository::new()` in presentation layer (55 sites)

### Migration strategy

No migration period. AppState is deleted and Extensions are registered in a single commit. Every handler is updated to extract only the Extensions it needs. No dual-wielding, no split-brain.

## Why

1. **Eliminates God Objects entirely** — neither AppState nor AppServices survives. Zero aggregator structs.
2. **No split-brain** — single migration step, no period where two systems coexist.
3. **Precise dependency declaration** — each handler's signature documents exactly which use cases it depends on.
4. **Testability** — tests construct only the Extensions needed per handler, not a 15-field god object.
5. **Clean Architecture compliance** — presentation depends on traits (Arc<dyn Trait>), not concrete infrastructure.
6. **Idiomatic Axum** — Extension<T> is the framework's intended DI mechanism.

## Consequences

### Positive

- Zero God Objects in the presentation layer
- Handlers self-document their dependencies via type signature
- Test construction: mock one trait instead of building a 15-field struct
- Clean Architecture seam: presentation → application traits, not infrastructure
- No migration complexity — single atomic change

### Negative

- More `.layer()` calls in Router setup (one per use case + one per infrastructure concern)
- Requires discipline to not create "mega-Extensions" that bundle multiple use cases
- ~130 handlers to update (73 pool.clone() + 55 repo constructor + handler signatures)
- Server startup registration order must be correct for all Extensions

### Migration risk

- 130+ handler sites to change in a single commit — requires thorough test coverage
- Must verify no handler accidentally retains AppState reference after migration
- Integration tests must run against the new Extension-based Router
