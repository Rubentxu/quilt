# DRAFT ADR: MigrationEngine — Collapse Generics to Arc<dyn Trait>

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-06-13
- **Source:** Q007 (Pass 1)

## Context

`MigrationEngine` is currently generic over three repository traits:

```rust
pub struct MigrationEngine<PR: PageRepository, BR: BlockRepository, PropR: PropertyRepository> {
    page_repo: Arc<PR>,
    block_repo: Arc<BR>,
    property_repo: Arc<PropR>,
}
```

This forces every call site to construct concrete types (`SqlitePageRepository::new(pool.clone())`) because Rust generics require `Sized` — you cannot write `MigrationEngine<dyn PageRepository, ...>`. The migration handler in `crates/quilt-server/src/handlers/migration.rs` therefore uses `pool.clone()` three times to construct concrete repos, bypassing the `FromRef<AppState>` extraction pattern used by every other handler in the codebase.

The `?Sized` threading approach (relaxing the `Sized` bound on the generic parameters) was considered but rejected — it adds complexity without addressing the root cause: the generics are unnecessary because MigrationEngine never needs monomorphization benefits (no per-type inlining, no zero-cost abstraction gain for a migration tool).

## Decision

**Delete the three generic parameters on MigrationEngine. Replace with `Arc<dyn Trait>` fields.**

Before:
```rust
pub struct MigrationEngine<PR: PageRepository, BR: BlockRepository, PropR: PropertyRepository> {
    page_repo: Arc<PR>,
    block_repo: Arc<BR>,
    property_repo: Arc<PropR>,
}
```

After:
```rust
pub struct MigrationEngine {
    page_repo: Arc<dyn PageRepository>,
    block_repo: Arc<dyn BlockRepository>,
    property_repo: Arc<dyn PropertyRepository>,
}
```

The handler then extracts repositories directly from AppState via `FromRef`:

```rust
async fn migrate_md_import(
    Extension(page_repo): Extension<Arc<dyn PageRepository>>,
    Extension(block_repo): Extension<Arc<dyn BlockRepository>>,
    Extension(property_repo): Extension<Arc<dyn PropertyRepository>>,
) -> ... {
    let engine = MigrationEngine::new(page_repo, block_repo, property_repo);
    ...
}
```

All three `SqliteXxxRepository::new(pool.clone())` sites in the migration handler are deleted.

## Why

1. **Root cause elimination**: The generics caused the `pool.clone()` problem. Removing them removes the problem, rather than working around it with `?Sized` bounds.

2. **Consistency**: Every other handler in quilt-server extracts `Arc<dyn Repository>` via `Extension<T>` + `FromRef`. MigrationEngine was the only outlier.

3. **No performance cost**: MigrationEngine is not in a hot path — it runs once per import request. Dynamic dispatch overhead is negligible compared to disk I/O and SQLite operations.

4. **Unblocks two fixes simultaneously**: Collapsing generics enables injecting the real `property_repo` (fixing Q006 EmptyPropertyRepo bug) AND eliminates `pool.clone()` sites (fixing Q001), in one structural change.

5. **Simpler code**: Three generic parameters with bounds disappear from every `impl` block, test helper, and call site. The type signature shrinks without losing any capability.

## Consequences

- **CONTEXT.md**: Document that MigrationEngine uses dynamic dispatch via `Arc<dyn Trait>`. Document the `FromRef` extraction pattern for handlers.
- **ADR**: This is the ADR. Promotes to numbered ADR after implementation.
- **API**: MigrationEngine::new signature changes. All callers (handler + tests) must pass `Arc<dyn T>` instead of `Arc<T>`.
- **Persistence**: No impact.
- **Tests**: `migration_tests.rs` already constructs engines with `Arc<InMemoryPageRepo>` etc. — these satisfy `Arc<dyn PageRepository>` trivially via coercion. Test changes are minimal.
- **Security**: No impact.
- **Observability**: No impact.
