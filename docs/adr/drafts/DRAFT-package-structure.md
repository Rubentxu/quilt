# DRAFT ADR: Package Structure — 3-Layer Hierarchy with Boundary Enforcement

- **Status:** draft (auto-generated by auto-grill-loop)
- **Date:** 2026-05-30
- **Source:** Q007-P1 (Pass 1)

## Context

Quilt's React frontend has a hybrid architecture with a strict WASM boundary (see DRAFT-hybrid-wasm-boundary): Rust/WASM for domain logic, TypeScript/React for UI shell. Without physical enforcement, this boundary exists only in documentation — code will drift toward tight coupling between the WASM bridge and React components.

The package structure must:
- Physically separate infrastructure (WASM bridge) from domain features from shared utilities
- Enforce the WASM boundary at build time (not just in documentation)
- Map features 1:1 to Logseq domain concepts for AI agent navigability
- Support the testing strategy layers (see DRAFT-testing-strategy-update)

## Decision

**3-layer hierarchy with ESLint import rule boundary enforcement.**

```
src/
├── core/           # Infrastructure layer
│   ├── wasm/       # get_state(), dispatch(), WASM module loader
│   └── router/     # TanStack Router configuration
├── features/       # Domain feature layer (maps 1:1 to Logseq concepts)
│   ├── outliner-tiptap/   # Block tree editing (Tiptap)
│   ├── editor-cm6/        # Embedded code blocks (CM6)
│   ├── journals/           # Daily journal navigation
│   ├── sidebar/            # Navigation sidebar
│   └── properties/         # Block property panel
├── shared/         # Cross-cutting layer
│   ├── types/      # Generated ts-rs types
│   ├── i18n/       # Internationalization
│   ├── telemetry/  # Observability (metrics, logging)
│   └── components/ # Shared UI components (buttons, modals, etc.)
└── pages/          # Route-level page compositions
```

**ESLint import rules (boundary enforcement):**

| From | Can import | Cannot import |
|------|-----------|---------------|
| `features/*` | `core/*`, `shared/*`, sibling features | — |
| `shared/*` | — | `core/*`, `features/*` |
| `core/*` | `shared/*` (types only) | `features/*` |
| `pages/*` | `features/*`, `core/*`, `shared/*` | — |

**Violations fail CI.** The boundary is physically enforced, not aspirational.

## Why

1. **Directory structure is architecture enforcement**: Without physical boundaries, Q002-P1's WASM boundary is aspirational. ESLint import rules make violations fail CI — architecture is enforced, not wished for.

2. **Core/features/shared separation**: Infrastructure (WASM bridge, routing) is distinct from domain features (outliner, journals) which is distinct from utilities (types, i18n, shared components). This prevents infrastructure code from becoming entangled with domain logic.

3. **Features map to domain concepts**: Each feature folder corresponds to a Logseq concept (outliner, journals, sidebar, properties). AI agents navigating the codebase can reason about the domain by reading the feature directory structure — no need to infer domain boundaries from code.

4. **Shared layer isolation**: Shared utilities cannot import from features or core — they are pure libraries with no domain knowledge. This prevents shared code from becoming a dumping ground for cross-cutting concerns.

## Consequences

- **ESLint configuration**: Custom import rules must be defined and enforced in CI.
- **Feature isolation**: Features can import from each other (e.g., outliner-tiptap imports from properties), but circular dependencies between features should be avoided.
- **Monorepo question**: Should each feature have its own `package.json` (Nx/turborepo) or stay as a single package with directory conventions? This is a follow-up decision.
- **AI navigability**: Feature directories named after domain concepts improve AI agent codebase understanding.
