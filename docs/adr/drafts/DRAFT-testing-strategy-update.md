## Context

ADR-0010 defines Quilt's current testing strategy: Rust unit tests, integration tests, and insta snapshot tests. The React migration introduces new testing dimensions:

- React component tests
- WASM↔React interop (the critical path — if data doesn't flow across this boundary, nothing works)
- Migration correctness (Leptos behavior vs React behavior during transition)
- E2E tests for the new frontend
- Visual regression for UI parity

The WASM↔React boundary is the single most important testing surface — it's the critical path that, if broken, collapses the entire architecture (see DRAFT-frontend-migration-strategy).

## Decision

**Preserve ADR-0010 layers. Add three new testing layers. Correct E2E gate metric.**

### Testing Layers

| Layer | Technology | Scope | Status |
|-------|-----------|-------|--------|
| **WASM domain tests** | Rust (cargo test) | Outliner core, parser, query DSL | **PRESERVED** — unchanged from ADR-0010 |
| **WASM snapshot tests** | insta (Rust) | Output correctness, regression | **PRESERVED** — unchanged from ADR-0010 |
| **React component tests** | Vitest + React Testing Library | Component rendering, user interactions | **NEW** |
| **WASM↔React interop contracts** ⚠️ | Vitest + WASM loader mock | get_state() ↔ component, dispatch() ↔ state mutation | **NEW — CRITICAL PATH** |
| **Migration correctness suite** | Vitest (temporary) | Leptos output vs React output parity | **NEW — TEMPORARY** |
| **Visual regression** | Playwright screenshots (advisory) | UI visual parity, not blocking | **NEW — ADVISORY** |
| **E2E smoke tests** | Playwright | Critical user journeys end-to-end | **UPDATED** |

### Gate Metrics

| Gate | Metric | Rationale |
|------|--------|-----------|
| **E2E smoke** | 100% pass rate | All critical journeys pass. NOT code coverage. |
| **Interop contracts** | 100% pass rate | WASM↔React data flow verified |
| **Migration suite** | 100% pass rate | Leptos behavior matches React behavior |
| **Code coverage** | 80% (hygiene) | Advisory metric — not a gate. Monitored, not blocking. |

### Tools

| Tool | Role |

## Why

1. **Critical path first**: The WASM↔React boundary is the single point of failure for the entire architecture. If `get_state()` returns wrong data or `dispatch()` doesn't mutate state, nothing else matters. Interop contract tests verify this boundary explicitly and are mandatory (gate condition).

2. **Wrong metric corrected**: ADR-0010 defined E2E gate as code coverage percentage. Coverage is a hygiene metric — 80% coverage can pass while all smoke tests fail. The correct gate metric is smoke test pass rate: every critical user journey must work. Coverage is monitored but not blocking.

3. **Migration needs verification**: During the Leptos→React transition, the old and new implementations coexist. Migration correctness tests compare Leptos output vs React output for the same inputs — verifying parity is achieved before switchover. These tests are temporary (deleted after Phase 3 cleanup).

4. **Visual regression is advisory**: Screenshot comparison catches visual drift but is fragile (pixel differences from font rendering, OS differences). It's advisory — signals problems without blocking CI.

5. **WASM domain tests preserved**: The outliner core tests in Rust are unchanged. They continue to verify domain logic correctness independently of the UI layer.

## Consequences

- **ADR-0010**: Updated with new layers and corrected metrics.
- **Interop test harness**: Must be built — mock WASM module loader for `get_state()` and `dispatch()` contract verification.
- **Migration suite**: Temporary test suite that is deleted after Phase 3 cleanup. Must be clearly marked as temporary in code and documentation.
- **E2E smoke test list**: Must be defined — list of critical user journeys that constitute "the app works."
- **Visual regression**: Screenshot baselines stored in repo. Advisory failures don't block CI but are reported.
