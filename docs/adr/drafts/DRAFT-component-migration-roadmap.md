## Context

Nested tree virtualization is a key concern — the outliner displays blocks with indent/outdent at arbitrary nesting depth. Not all virtualization libraries support tree structures.

### Virtualization

**TanStack Virtual REJECTED for nested tree outliner.** It is designed for flat lists — no tree indentation awareness, no collapse/expand state management, no indentation-aware scroll positions.

## Why

1. **Evidence over guessing**: "Most PORT" without per-component source analysis is a wish, not a plan. Each component was analyzed against Q002-P1's boundary: framework-coupled code is not portable; algorithmic code is.

2. **Graph View is algorithmic**: The 469-line `force_graph.rs` implements D3-force layout and node/edge management — pure algorithm, no Leptos view coupling. It's a strong PORT candidate.

3. **Nested trees ≠ flat lists**: TanStack Virtual (and most virtualization libraries) assumes a flat list of items with uniform height. Block outliners have variable block height (text content, nested children, collapsed state) and indent-based positioning. Recommending a flat-list virtualization library for a tree is a category error.

4. **Realistic costing**: BUILD items need person-week estimates to inform the migration timeline and resource planning. "6-12 person-weeks" is a realistic range for complex components (sidebar with drag-resize, tree navigation, panels).

5. **Dependency ordering matters**: Sidebar blocks Journals (journals navigation goes through sidebar). Search depends on query DSL placement (unresolved). Ordering BUILD work by dependency prevents blocked work.

## Consequences

- **Migration timeline**: BUILD components (sidebar, journals, properties, search) total 22-32 person-weeks. PORT components (graph view, CM6 code blocks) total 4-6 person-weeks.
- **Virtualization**: TanStack Virtual not used. react-virtuoso or custom solution must be evaluated and POC'd.
- **Query DSL**: Search component is blocked until query DSL placement is decided (Q002-P1 follow-up).
- **Person-week estimates**: Rough estimates — team review and refinement needed before sprint planning.
