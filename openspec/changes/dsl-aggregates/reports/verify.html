Corrections Applied (Iteration 1)

J1-1/J2-1: Aggregate hardcoded $.count — CRITICAL

  • Problem: Avg/Sum/Min/Max aggregate functions used hardcoded $.count instead of the group_by property path
  • Fix: Changed to use prop_path variable which contains json_extract(properties, '$.{group_by}')
  • Before: AVG(CAST(json_extract(properties, '$.count') AS REAL))
  • After: AVG(CAST({prop_path} AS REAL))

J1-3: percentile_cont PostgreSQL syntax — CRITICAL

  • Problem: percentile_cont(...) is PostgreSQL-only, not valid SQLite syntax
  • Fix: Replaced with SQLite-compatible ROW_NUMBER() subquery pattern for median and percentile
  • Before: percentile_cont(0.5) WITHIN GROUP (ORDER BY {prop})
  • After: (SELECT val FROM (SELECT {prop} as val, ROW_NUMBER() OVER (ORDER BY {prop}) as rn, COUNT(*) OVER () as total FROM blocks b WHERE {prop} IS NOT NULL) WHERE rn = CAST(total * {frac} AS INTEGER))

J1-6/J2-2: Stats WHERE clause contains aggregate function — CRITICAL

  • Problem: build_where() for Stats variant returned {prop} AND {STDDEV_POP(...)} — aggregate functions in WHERE are invalid SQL
  • Fix: Removed Stats from build_where(); Stats is only handled in build_sql() which generates complete SELECT. Added panic with clear message if someone calls build_where() with Stats
  • Result: build_where() now only handles Aggregate and GroupBy (for nested queries), Stats is exclusively handled in build_sql()
