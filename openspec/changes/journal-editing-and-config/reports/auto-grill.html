2. Auto-Resueltos

✅ Q1: Es JournalDay Display → Page name la misma cadena?

Si. Page::new_journal usa day.to_string() para name y title. Como Display produce "YYYY-MM-DD", el nombre de pagina ES el string ISO. Cambiar Display rompe los lookups por nombre.

Accion: Agregar to_formatted() separado de Display. Display se mantiene ISO para DB.

Confidence: 0.95 | Method: CogniCode + code reading

✅ Q2: quilt-http compila con SqliteSettingsRepository roto?

No compila, pero no importa: quilt-http NO esta en el workspace (ver Cargo.toml). El import roto no genera errores de build.

Accion: Si se necesita quilt-http, hay que implementar SqliteSettingsRepository primero.

Confidence: 1.0 | Method: Cargo.toml workspace members + grep

✅ Q3: outliner_block.rs es codigo activo?

No. No aparece en components/mod.rs. Es codigo muerto. Ademas tiene un bug: llama update_block con 6 parametros pero la funcion acepta 2.

Accion: Ignorar outliner_block.rs para este cambio. No afecta la arquitectura activa.

Confidence: 1.0 | Method: mod.rs reading + grep

✅ Q4: Block on_save pierde datos al no llamar al server?

No necesariamente. El PageOutliner coordina el historial de undo/redo localmente. La persistencia server probablemente se maneja por otro camino (ej: un watcher o sync periodico). Esto es un PATRON INTENCIONAL de optimistic updates — pero necesita verificacion.

Accion: Este cambio NO debe modificar el patron de persistencia. Solo agregar contexts a JournalView.

Confidence: 0.7 | Method: heuristic | Riesgo: medio si el patron no es intencional
